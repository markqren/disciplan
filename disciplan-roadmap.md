| FEA-34 | **Multi-Currency: CAD as Source of Truth for TD Accounts** | Feature | **High** | TD-prefixed accounts (TD Chequing, TD Savings, TD TFSA, TD Visa, TD Debit) should treat CAD as the authoritative amount, with USD as a floating conversion — not the current approach where `amount_usd` was computed once at import using a fixed 0.73 rate. **Phased approach:** **Phase 1 — Configurable FX rate + Balance Sheet:** Add a `cad_usd_rate` setting (Supabase settings table or app constant, default 0.73). Update `get_ledger_balances()` RPC to return both native currency balance and USD-converted balance for TD accounts (native = CAD, converted at current rate). Balance Sheet displays USD but hover tooltip shows the real CAD balance. **Phase 2 — IS / Tags / Charts:** Accruals for TD transactions should use `original_amount` (CAD) converted at the current rate instead of stale `amount_usd`. `daily_cost` for CAD transactions recalculated dynamically or conversion applied at display time. **Phase 3 — Live FX:** Pull from a free FX API (e.g. exchangerate.host) instead of manual rate. **Database notes:** The existing `original_amount` and `currency` columns already store CAD values — the issue is that `amount_usd` was baked once and never updates. The one-time ledger adjustment for TD Chequing should be stored in CAD so it stays correct when the FX rate changes. |
| FEA-38 | **Balance Sheet: Working Capital Section + Credits & Transfers Promotion** | Feature | **Medium** | Restructure the Balance Sheet Liabilities panel from the current layout (Credit Cards → Other Liabilities → Credits & Transfers nested underneath) into three distinct sections: **Credit Cards** (unchanged — Apple, AMEX, Chase cards, Bilt, Uber, TD Visa, etc. with `account_type = 'credit'`), **Working Capital** (new section replacing "Other Liabilities" — accounts where money flows through with a real float/balance: Venmo, Venmo Credit, and eventually Rakuten, Splitwise), and **Credits & Transfers** (promoted to its own standalone section, no longer nested under Other Liabilities — the sub-ledger of IOUs: Home, Tony, Kevin, HSA, FSA, etc. from `get_credit_balances()` RPC). **Implementation:** (1) Add new `account_type = 'working_capital'` value. Reclassify Venmo and Venmo Credit from `liability` → `working_capital` in accounts table. (2) Update `renderBS` to render three groups under Liabilities: `acctGroup("Credit Cards", byType.credit)`, `acctGroup("Working Capital", byType.working_capital)`, then Credits & Transfers as its own top-level expandable section (same UI as now but no longer visually nested inside another group). (3) Update `byType` initialization to include `working_capital:[]` and ensure KPI totals include working_capital in liabilities sum. (4) Future accounts like Splitwise (FEA-29) also get `working_capital` type. **Rationale:** "Other Liabilities" was a catch-all that mixed genuinely different account types. Working Capital accounts are intermediary float accounts (money in transit, pending payouts). Credits & Transfers are personal IOUs tracked via the Transfer/credit system. Separating them makes the Balance Sheet more readable and sets up Rakuten cashback tracking (FEA-37). **Dependencies:** FEA-35 (Credits & Transfers sub-ledger, already done). |
| FEA-29 | **Reimbursement & Splitwise Integration** | Feature | **High** | Two-part feature to streamline reimbursement tracking. **Part A — One-Click Reimburse on Ledger Items:** Add a "Reimburse" action button on any ledger transaction row (or in the edit modal from FEA-21). Clicking it opens a reimbursement form pre-populated with: the original transaction's amount, a split % selector (default 50%, with common presets: 50%, 33%, 25%, custom), the person to reimburse (dropdown of friends: Tony, Kevin, Aditya, Home, etc. derived from existing Transfer credit names in the data), and payment type defaulting to "Splitwise". On submit, it creates a new offsetting transaction: negative amount for the reimbursed portion, category matching the original, description like "Reimbursed - [original description]", payment type = "Splitwise", same service period as original. This replaces the old manual pattern of using Transfer payment type with credit names like "Kevin", "Home", "Cornerstone" etc. **Part B — Splitwise API Sync:** Splitwise has a free Self-Serve API (dev.splitwise.com) that supports OAuth2 and provides `getExpenses()` with date filters, plus `getFriends()` and `getGroups()`. Build a sync feature that: (1) authenticates with Splitwise via OAuth2 (register app at secure.splitwise.com/oauth_clients), (2) fetches expenses where the user owes or is owed money, (3) maps Splitwise expenses to Disciplan transactions — expenses you paid get the actual category + a Splitwise reimbursement credit for others' shares, expenses others paid show as your owed share under the "Splitwise" payment type, (4) maintains a "Splitwise" account in the balance sheet that tracks your net balance (what you're owed minus what you owe), which should stay in sync with your actual Splitwise balance. A JavaScript SDK exists (npm: splitwise) but since this is a single HTML file, the REST API can be called directly with fetch(). Rate limits are conservative so sync should be periodic (manual trigger or daily), not real-time. Supersedes FEA-16. **Note:** The "Splitwise" payment type account should be classified as `account_type = 'working_capital'` in the accounts table so it appears under the "Working Capital" section of the Balance Sheet alongside Venmo and Rakuten (see FEA-38). |

This accrues the pending balance — Rakuten now owes Mark money, shown as a positive balance in the Working Capital section of the Balance Sheet. (2) When the quarterly Big Fat Check arrives (~every 3 months), create a settlement pair: "Rakuten Payout" = positive amount on Rakuten (zeroing out the balance) + "Rakuten Payout Deposited" = negative amount on Chase Chequing (cash received). When an email-ingested transaction is created, attempt to match it to a related existing transaction: cashback → original purchase (merchant fuzzy match + date proximity 0-90 days), Venmo received → the expense it reimburses (amount match + description), bank alert → existing imported CSV transaction (amount + date exact match for dedup/confirmation). Clicking one linked transaction highlights or scrolls to its partner. In the edit modal, show a "Linked to:" reference with the partner's description and amount. **Phase 5 — In-App Email Review Queue:** New section in the Import tab (or its own tab) showing pending emails from `email_inbox` with status = pending. Each row shows sender, subject, parsed amount, suggested transaction preview. User can approve (creates transaction), edit (opens pre-filled entry modal), or dismiss. Processed emails link back to the created transaction for audit trail. **Existing data patterns:** Rakuten cashback is currently manually entered as negative amounts on Chase Chequing (e.g. "Rakuten - Chewy" = -$16.50, "Rakuten - Sur la Table" = -$8.14), always placed adjacent to the corresponding purchase. Venmo transactions already exist as a payment_type but lack linking to the expenses they reimburse. **Dependencies:** FEA-10 (Auth) for securing the webhook endpoint, FEA-36 (CC Payment Automation) for the `related_transaction_id` linking pattern, FEA-38 (Working Capital section) for Rakuten account placement. **Complements:** FEA-14 (Cashback tracking/analytics), FEA-29 (Splitwise/reimbursement integration). |
